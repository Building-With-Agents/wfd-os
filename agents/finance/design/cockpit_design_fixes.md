### Sub-phase 1C — polymorphic drill content structure

This is the most important structural change. The current schema
(`eyebrow` / `title` / `summary` / `sections: [{title, rows: [{label,
value}]}]`) only supports label/value pairs. That's inadequate for
Finance (the Year Up placement history is crying out to be a chart)
and will be completely wrong for Recruiting (match score breakdowns,
student histories, job requirements won't fit).

Define and implement the following schema. Each drill entry now looks
like this:

```python
{
    "eyebrow": "Provider",                    # kicker, uppercase
    "title": "Year Up",                        # Instrument Serif headline
    "summary": "Full provider view · ...",    # one-line context
    "status_chip": {                          # optional, system metadata
        "label": "Early access",
        "tone": "watch"                       # good | watch | critical | neutral
    },
    "sections": [
        # ... mixed types, see below
    ],
    "actions": [                              # drill footer buttons
        {"label": "Open placements tab", "intent": "navigate", "target": "placements"},
        {"label": "Ask assistant about this", "intent": "chat", "prompt": "..."},
        {"label": "Export detail", "intent": "export"}
    ],
    "updated_at": "2026-04-17T10:42:00Z",     # for freshness display
    "source": "WSAC + QB + v3 reconciliation" # for source chip
}
```

Where `sections` is a list of **typed** content blocks:

```python
# Type: rows — keep the existing label/value pattern for simple lookups
{
    "type": "rows",
    "title": "Contract & Spend",
    "rows": [
        {"label": "Amended budget", "value": "$340,000"},
        {"label": "Paid to date", "value": "$132,500"},
        {"label": "Balance remaining", "value": "$207,500", "emphasize": True}
    ],
    "note": "Optional small-print context below the rows"  # optional
}

# Type: table — when the data has natural columns
{
    "type": "table",
    "title": "Placements",
    "columns": [
        {"key": "period", "label": "Period"},
        {"key": "placements", "label": "Placements", "align": "right", "numeric": True},
        {"key": "paid", "label": "Paid", "align": "right", "numeric": True}
    ],
    "rows": [
        {"period": "Q1'24", "placements": 16, "paid": "$27,000"},
        {"period": "Q2'24", "placements": 15, "paid": "$24,000"},
        # ...
        {"period": "Total", "placements": 97, "paid": "$213,000", "total": True}
    ]
}

# Type: chart — for trends and comparisons
{
    "type": "chart",
    "title": "Placements over time",
    "chart_type": "bar",                      # bar | line | area
    "x_axis": {"key": "period", "label": "Quarter"},
    "y_axis": {"key": "placements", "label": "Count"},
    "data": [
        {"period": "Q1'24", "placements": 16},
        # ...
    ],
    "reference_lines": [                       # optional, for targets/thresholds
        {"value": 81, "label": "Target", "tone": "good"}
    ]
}

# Type: prose — when the answer is narrative
{
    "type": "prose",
    "title": "Contract status",
    "body": "Contract terminated by ESD direction. 56 placements net
             of Q1 retraction, significantly below the 90+ target.
             Cumulative cost per placement at $7,068 exceeds the
             amber/red threshold band..."
}

# Type: verdict — editorial stance with emphasis
{
    "type": "verdict",
    "tone": "watch",                           # good | watch | critical
    "headline": "Ada is cost-effective — but CFA has underpaid.",
    "body": "Ada delivered 110 placements against $132,500 paid,
             yielding a true CPP of $1,052 — the second-lowest in
             the cohort. The contract balance remaining ($207,500)
             suggests CFA owes another ~$72,500 in unpaid invoices
             that haven't cleared the tracker."
}

# Type: timeline — for sequenced events
{
    "type": "timeline",
    "title": "Contract history",
    "events": [
        {"date": "2024-01-01", "title": "Contract start", "tone": "neutral"},
        {"date": "2025-11-15", "title": "Amendment 1 approved", "tone": "good"},
        {"date": "2026-03-31", "title": "Contract close", "tone": "neutral"}
    ]
}

# Type: action_items — inline open action items for this entity
{
    "type": "action_items",
    "title": "Open action items (2)",
    "items": [
        {"priority": "HIGH", "owner": "Krista",
         "text": "Confirm Q4 '25 invoice paid or outstanding"},
        {"priority": "HIGH", "owner": "Ritu / Andrew",
         "text": "Clarify -19 retraction in Q1 2026"}
    ]
}
```

1. **Schema definition.** Create a Pydantic model (or TypedDict) in
   `cockpit_data.py` for the drill content schema. Validate every
   registry entry against it at build time. Fail loudly if any drill

2. **Renderer per section type.** Each section type gets its own
   rendering function. In the HTML version, these emit the appropriate
   markup (rows stays as now; table uses `table.data`; chart uses
   small inline SVG; prose uses a paragraph; verdict reuses the
   `.verdict` component; timeline uses a simple date-left/content-right
   grid; action_items reuses the existing action-row style).

3. **Migrate existing drills.** The seven category drills and eleven
   decision drills can stay as `rows` type — they're fine as is. The
   provider drills need upgrading:
   - Contract & Spend → `rows` (unchanged)
   - Placements → **change from rows to table** (period, placements, paid columns)
   - Cost Analysis → **change to verdict** for providers in red/amber bands,
     rows for green
   - Open Action Items → **change to action_items type**
   Add a new section per provider: a `chart` type showing placements over
   time, same data as the table but visualized.

4. **Drop the repeated boilerplate.** The sentence "Once Class tracking
   and production QB sync land, this drill will also list every
   transaction in the category" appears in seven category drills.
   Promote to `status_chip: {"label": "Transaction detail pending", "tone": "watch"}`.
   Don't repeat as prose.

5. **Consolidate color-band logic into data.** The cockpit currently
   computes color bands in two places: `cockpit_data.py` assigns
   `threshold_band` strings ("Green", "Amber", "Red") on some rows,

   ```
   {% set color = '#1F6B5C' if pct >= 100 else '#B8821B' if pct >= 75 else '#B8462E' %}
   ```
