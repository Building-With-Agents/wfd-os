# WFD-OS v1.3 Specification — Compliance Engine + Finance Cockpit Multi-Tenant Integration & Audit Readiness Expansion

**Status:** Draft, in active design.
**Spec owner:** Ritu Bahl (product).
**Implementation owner:** Gage (handoff in progress).
**Strategic context:** Path 2 — multi-tenant platform serving the intermediary segment. CFA and Borderplex are first two tenants; Waifinder owns the platform.

## Purpose

v1.3 has two intertwined goals:

1. **Integration:** wire the v1.2 compliance engine and finance cockpit onto Gary's `wfdos-common` shared infrastructure, resolving the architectural debt accumulated during the v1.2 build, and onboarding CFA as a tenant.
2. **Expansion:** convert the four placeholder dimensions in the audit readiness tab into computed dimensions, plus add PBC tracker and audit firm engagement tracking as new compliance surfaces.

Integration work must precede dimension expansion. Building new dimensions on the pre-integration architecture would create rework when the integration lands.

This spec describes the entire v1.3 scope. Implementation will be sequenced into sub-projects (v1.3.0 integration foundation, v1.3.1-v1.3.4 dimensions, v1.3.5-v1.3.6 cross-cutting). Each sub-project gets its own implementation focus, but all are governed by this single spec.

## 1. Path 2 architectural north star

**Principle:** Every architectural decision in v1.3 is made with multi-tenant deployability as a first-class constraint, not as future tech debt to address later.

**What this means in practice:**

- No tenant-specific assumptions in compliance engine or cockpit code. CFA-specific values (grant ID K8341, role-to-name mapping, program-specific compliance rules) become per-tenant configuration, not hardcodes.
- Connector implementations parameterize on tenant, not on data source. A SharePoint reader doesn't know "this is CFA's SharePoint" — it knows "this is the SharePoint configured for tenant X."
- Compliance rules are configuration-driven where the underlying regulation supports it. A program-specific compliance supplement (GJC for CFA, IFLE for Borderplex if applicable) is loaded per tenant, not baked into module code.
- The compliance engine's data model treats `tenant_id` as a required scoping dimension on every domain table that holds operational data. Reference data (regulations, standard rule definitions) may be shared across tenants when the regulatory text is identical.
- The audit readiness tab renders tenant-scoped data. A user authenticated as a CFA staff member sees CFA's compliance state. A future Borderplex staff member sees Borderplex's.

**What this does NOT mean:**

- Premature abstraction. Where v1.3 needs CFA-specific behavior to ship the first dimension, that behavior is implemented as a documented exception with an explicit "generalize when second tenant adopts" note. Speculative generality across all four dimensions on day one would slow delivery without proportionate value.
- Speculative connectors. Each connector is built for a real tenant's real data source. Building a generic "SharePoint connector that works for any tenant" before any second tenant uses it is YAGNI. Build for CFA first; harden for second tenant when Borderplex actually needs the dimension.
- Tenant-aware UI components. The cockpit is rendered server-side per tenant; UI components don't need tenant-awareness, only the data layer does.

**Tenant model:**

Per Gary's `wfdos-common` design (see discovery report dated 2026-04-23), tenants are represented as string slugs:
- `waifinder-flagship` (platform owner, internal use)
- `borderplex` (intermediary client)
- `cfa` (intermediary client — to be added in v1.3.0)

Both CFA and Borderplex fall under the **intermediary** segment. They differ from each other in operational details (grants administered, program scope, payment terms with Waifinder) but are architecturally equivalent. There is no "CFA-special" path in the codebase.

## 2. wfdos-common integration plan

v1.2 was built before `wfdos-common` existed in usable form. This section enumerates the modules in `wfdos-common` and the integration approach for each.

### 2.1 db — per-tenant SQLAlchemy engine factory

**v1.2 state:** Compliance engine creates its own SQLAlchemy engine via local connection-string assembly. No tenant scoping.

**v1.3 target:** Replace local engine creation with `wfdos_common.db.register_tenant("cfa", connection_url)` at startup. All session creation goes through `wfdos_common.db.session_scope(tenant_id="cfa")` (server-side) or `db_session(request)` FastAPI dependency (request-scoped).

**Migration approach:**
- Replace existing `get_db_session()` calls with `wfdos_common.db.session_scope(tenant_id=...)`
- Replace FastAPI dependency in routes with `db_session(request)` which reads `request.state.tenant_id`
- Remove local `engine.py` once all call sites migrated

**Open question:** Where does the per-tenant connection URL come from at startup? Hardcoded environment variables (one per tenant) is the simplest answer; a tenants table in a control-plane DB is the more scalable answer. Defer decision to v1.3.0 implementation; ask Gary what `wfdos-common` expects.

### 2.2 auth — magic-link + session middleware + role deps

**v1.2 state:** No auth on FastAPI routes. Anyone with network access to the engine or cockpit can call any endpoint.

**v1.3 target:** Install `wfdos_common.auth.SessionMiddleware` on both compliance engine and cockpit. Gate routes via `wfdos_common.auth.require_role("staff")` or appropriate ladder position. Mount `build_auth_router()` at `/auth/*` for magic-link login.

**Role mapping for compliance/cockpit access:**
- `admin` — Ritu (CFA), Waifinder platform admins
- `staff` — Krista (finance), Bethany (placements), Gage (developer with operational access)
- `workforce-development` — provisional, may not be needed for v1.3 surfaces
- `student` — explicitly NOT allowed access to compliance/cockpit

**Tier decorators:** routes that perform LLM calls should use `@llm_gated(...)` to give Gary's infrastructure the chance to enforce per-tier rate limits.

### 2.3 errors — typed APIError hierarchy

**v1.2 state:** Ad-hoc error responses. Some routes return `HTTPException`, some return JSON dicts with error fields, some let exceptions bubble.

**v1.3 target:** Replace ad-hoc patterns with `wfdos_common.errors.APIError` subclasses. Install `install_error_handlers(app)` on both services. Standardizes error response shape across all WFD-OS surfaces.

**Migration mapping (compliance engine):**
- `404` responses → `NotFoundError`
- `400` validation responses → `ValidationFailure`
- QB OAuth failures → `UnauthorizedError`
- Engine-internal failures with retry potential → `ServiceUnavailableError`

### 2.4 llm — provider-agnostic adapter with fallthrough

**v1.2 state:** Verdict generator (cockpit) and explanation endpoint (engine) call Gemini directly via `google.generativeai`. Bare provider, no fallthrough.

**v1.3 target:** Migrate both to `wfdos_common.llm.complete(messages, tier=..., system=...)`. Get Azure OpenAI as default with Anthropic and Gemini as fallthroughs. Provider failures become transparent — cockpit and engine no longer need to handle provider-specific errors.

**Migration scope:**
- `agents/finance/verdict_generator.py` — replace `genai.GenerativeModel(...).generate_content(...)` with `wfdos_common.llm.complete(...)`
- `agents/grant-compliance/.../explain_flag.py` (or wherever the explain endpoint lives) — same pattern
- Remove `google-generativeai` as a direct dependency once both call sites migrated; `wfdos-common` carries provider SDKs internally

**Tier selection:** Verdict generation is a low-stakes editorial task. Use `tier="default"` (cheap, fast). Explanation generation is similarly editorial. Reserve `tier="reasoning"` for any future workflows that need deeper LLM reasoning (e.g., draft-time-effort-certification generation in v1.3.3).

### 2.5 graph — MS Graph client (SharePoint)

**v1.2 state:** Compliance engine has its own SharePoint reader code in `agents/grant-compliance/.../graph_helper.py` (or similar). Pre-`wfdos-common`.

**v1.3 target:** Migrate to `wfdos_common.graph`. Per discovery report, this is largely a 1:1 import migration since `wfdos-common`'s Graph submodule is itself a migration of `agents/graph/sharepoint.py`.

**Concern:** `wfdos-common.graph` uses a single global `ClientSecretCredential(tenant_id=AZURE_TENANT_ID, ...)` — not per-tenant. For CFA-only deployment in v1.3, this works (CFA's SharePoint is the only one being read). For multi-tenant production, each tenant needs its own credentials.

**v1.3 decision:** Accept the single-global-credential limitation for v1.3.0 integration. Document as known constraint. Per-tenant Graph credentials become v1.4 work or a follow-up Gary discussion.

### 2.6 logging — structlog with request context

**v1.2 state:** Mixed `print()`, `logging.info()`, ad-hoc patterns.

**v1.3 target:** Adopt `wfdos_common.logging` for structured logs with request context (tenant_id, request_id, actor). Greatly improves observability for any production debugging.

### 2.7 tenancy — BrandConfig registry

**v1.3 target:** Register CFA as a brand at startup via `wfdos_common.tenancy.register_brand(BrandConfig(slug="cfa", display_name="Computing for All", ...))`. Document brand fields needed (display name, primary color, logo path if cockpit becomes brand-aware in UI).

### 2.8 agent — Agent ABC + ToolRegistry

**v1.3 decision:** Defer migration of existing assistant agents (`agents/assistant/finance_agent.py`, etc.) onto the Agent ABC pattern. Per discovery report, this migration is itself a follow-up in `wfdos-common` (#26). Compliance engine and cockpit don't currently expose conversational agents that would benefit from the ABC. Revisit in v1.4.

## 3. Tenant onboarding mechanics for CFA

CFA must be added as a tenant in `wfdos-common` before any v1.3 work can proceed. This section specifies what onboarding entails.

### 3.1 Registration steps

Required actions, in order:

1. **Provision per-tenant database.** A schema or database for CFA's compliance data. Approach depends on Gary's `wfdos-common.db` model — schema-per-tenant in a shared DB, or DB-per-tenant. Decision deferred to v1.3.0 implementation; ask Gary.
2. **Register tenant in `wfdos_common.db`.** Call `register_tenant("cfa", connection_url_for_cfa)` at compliance engine and cockpit startup.
3. **Register brand in `wfdos_common.tenancy`.** Call `register_brand(BrandConfig(slug="cfa", display_name="Computing for All", ...))`.
4. **Seed CFA-specific reference data:**
   - Grant K8341 record (per CFA's grants table once it exists)
   - GJC compliance supplement reference (program-specific compliance rules)
   - Subrecipient seed data (the providers — Ada, Year Up, Vets2Tech, etc.)
   - User-actor mapping (krista@cfa.org → role staff, ritu@computingforall.org → role admin, gage@... → role staff or admin per Ritu's call)
5. **Migrate v1.2 compliance data.** Existing flags, transactions, attachments need to land in CFA's tenant scope. One-time data migration script.

### 3.2 Tenant resolution at request time

Per `wfdos_common.db.middleware.TenantResolver` (the authoritative middleware per Gary's design):
- Tenant resolved from `X-Tenant-Id` HTTP header (set by nginx edge proxy at trust boundary), OR
- Host header → tenant mapping (e.g., `cfa.wfd-os.com` → tenant `cfa`), OR
- Default tenant via `WFDOS_DEFAULT_TENANT_ID` environment setting

For local CFA development: set `WFDOS_DEFAULT_TENANT_ID=cfa` in `.env`. For production: nginx routes per host header.

### 3.3 What "CFA as tenant" looks like to users

No user-visible change for CFA staff in v1.3. Krista and Bethany continue to see the cockpit and compliance engine as they did in v1.2; tenant scoping happens beneath the surface.

When Borderplex (or any other intermediary) becomes a tenant of the platform, their staff see their own data, their own brand, their own subrecipients. Same code path; different tenant scope.

## 4. Path 2 readiness debt resolution (from v1.2)

v1.2 was built with several CFA-specific shortcuts. Each must be parameterized for multi-tenancy in v1.3. This section enumerates them with the resolution approach.

### 4.1 Role-to-display-name mapping

**v1.2 state:** Hardcoded in `agents/finance/audit_dimension_display.py`:

```python
_ROLE_DISPLAY_NAMES = {"bookkeeper": "Krista", "executive_director": "Ritu", "program_operations": "Bethany · Gage"}
```

**v1.3 resolution:** Replace with per-tenant config lookup. Each tenant has a role-to-person mapping stored in their tenant config (likely as a JSON column on a `tenant_config` table or as part of the brand registration). Lookup signature becomes `display_name_for_role(role: str, tenant_id: str) -> str`.

### 4.2 Actor-to-display-name mapping

**v1.2 state:** Hardcoded in `agents/finance/audit_activity_labels.py`:

```python
_ACTOR_DISPLAY_NAMES = {"krista@cfa.org": "Krista", "ritu@computingforall.org": "Ritu", ...}
```

**v1.3 resolution:** Same pattern as 4.1. Per-tenant config maps actor identifiers (typically email addresses) to display names. Unknown actors fall back to local-part of email.

### 4.3 Hardcoded "K8341" references

**v1.2 state:** Several places assume the active grant is K8341. Documentation gap threshold ($2,500) is K8341-derived. T&E placeholder copy mentions K8341 explicitly.

**v1.3 resolution:** Introduce a `Grant` model in the compliance engine (per-tenant scoped). Per-tenant config specifies which grants are "active" for compliance purposes. Computations parameterize on grant or aggregate across all active grants for the tenant. Documentation gap threshold becomes a per-grant configuration.

### 4.4 GJC Compliance Supplement assumptions

**v1.2 state:** The 12 Subpart E rules in `unallowable_costs.py` are general 2 CFR 200 Subpart E. But GJC-specific compliance supplement requirements would also apply if surfaced.

**v1.3 resolution:** Introduce a `ProgramCompliance` reference data model. Each grant references a program (GJC for K8341, others for hypothetical other CFA grants or Borderplex grants). Program-specific compliance rules load per grant. Subpart E rules remain shared reference data.

### 4.5 Cockpit fallback dimension titles

**v1.2 state:** `_FALLBACK_DIMENSION_TITLES` in `cockpit_data.py` hardcodes the six audit dimensions for the engine-unreachable case.

**v1.3 resolution:** Cockpit fetches dimension metadata from engine on startup and caches it. Engine-unreachable fallback uses the cached metadata, not hardcoded literals. If the cockpit has never successfully reached the engine, falls back to a generic "Audit dimensions unavailable — engine has not yet been reached" message.

### 4.6 Test infrastructure gap on cockpit branch

**v1.2 state:** Cockpit branch has no pytest infrastructure. All cockpit-side test work was deferred.

**v1.3 resolution:** Set up pytest scaffolding on the cockpit branch (or its v1.3 successor) as part of v1.3.0 integration work. Backfill tests for the v1.2 cockpit-side modules (`audit_dimension_display.py`, `audit_activity_labels.py`, `verdict_generator.py`). Future cockpit work has tests.

### 4.7 No JS/TS test infrastructure

**v1.2 state:** No Jest/Vitest in `portal/student/package.json`.

**v1.3 resolution:** Lower priority than Python pytest. Defer to v1.4 unless Gage finds the gap blocks meaningful work.

## 5. Per-tenant credential vault for QB OAuth tokens

v1.2 deferred QuickBooks OAuth token encryption and refresh handling pending Gary's `wfdos-common` work. Discovery report confirms `wfdos-common` does NOT include a credential vault — magic-link and session tokens use `itsdangerous` HMAC signing, which is fine for auth cookies but inappropriate for tenant-credential storage.

The QB OAuth token storage problem remains. v1.3 must solve it.

### 5.1 Requirements

- **Encryption at rest.** OAuth refresh tokens are long-lived secrets that grant access to QuickBooks data. Cannot be stored plaintext.
- **Per-tenant isolation.** Tenant A's credentials must be cryptographically inaccessible from tenant B's request context. A breach of one tenant's session must not expose another tenant's credentials.
- **Refresh handling.** OAuth access tokens expire (typically 1 hour). Refresh tokens have longer lifetimes but also expire. Refresh logic must be automatic and transparent to calling code.
- **Audit trail.** Every credential access logs (tenant, actor, purpose, timestamp).

### 5.2 Architecture options

**Option A — Add to `wfdos-common`.** Cleanest for Path 2: every tenant gets the same credential primitive. Requires Gary's involvement and acceptance of the design. Likely the right long-term answer.

**Option B — Build inside grant-compliance, designed for promotion.** Build a `credentials.py` module inside the compliance engine that follows interfaces a future `wfdos-common.credentials` module would expose. When ready to promote, copy to `wfdos-common` with minimal API change.

**Option C — Live with plaintext through v1.3.** Document as known security debt. Defensible only if (a) production deployment doesn't yet exist, (b) database access is tightly controlled, (c) v1.4 commits to addressing.

**v1.3 recommendation:** Option B. Builds the primitive while keeping forward momentum on v1.3 integration. Gives Gary something concrete to react to if/when the primitive promotes to `wfdos-common`. Avoids blocking v1.3.0 on a Gary conversation.

### 5.3 Implementation sketch (Option B)

Module `agents/grant-compliance/.../credentials.py`:

- `class CredentialVault` — abstract or concrete; per-tenant scoped
- `vault.store_oauth_credentials(provider: str, access_token: str, refresh_token: str, expires_at: datetime, metadata: dict)` — stores encrypted; provider is e.g. `"qb_online"`
- `vault.get_oauth_credentials(provider: str) -> OAuthCredentials | None` — returns decrypted; raises `UnauthorizedError` if no creds stored or refresh fails
- `vault.refresh_oauth_credentials(provider: str) -> OAuthCredentials` — explicit refresh; called automatically by `get_oauth_credentials` if access token has expired

Encryption: Fernet (cryptography library), key per-deployment via environment variable. Future per-tenant keys would extend this.

Storage: new `tenant_credentials` table, scoped by `tenant_id` per the per-tenant DB model.

Audit: every store / get / refresh writes to the existing `audit_log` table (action types `credentials.store` / `credentials.access` / `credentials.refresh`).
