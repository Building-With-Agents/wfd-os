## 1. Path 2 architectural north star

**Principle:** Every architectural decision in v1.3 is made with multi-tenant deployability as a first-class constraint, not as future tech debt to address later.

**What this means in practice:**

- No tenant-specific assumptions in compliance engine or cockpit code. CFA-specific values (grant ID K8341, role-to-name mapping, program-specific compliance rules) become per-tenant configuration, not hardcodes.
- Connector implementations parameterize on tenant, not on data source. A SharePoint reader doesn't know "this is CFA's SharePoint" — it knows "this is the SharePoint configured for tenant X."
- Compliance rules are configuration-driven where the underlying regulation supports it. A program-specific compliance supplement (GJC for CFA, IFLE for Borderplex if applicable) is loaded per tenant, not baked into module code.
- The compliance engine's data model treats `tenant_id` as a required scoping dimension on every domain table that holds operational data. Reference data (regulations, standard rule definitions) may be shared across tenants when the regulatory text is identical.
- The audit readiness tab renders tenant-scoped data. A user authenticated as a CFA staff member sees CFA's compliance state. A future Borderplex staff member sees Borderplex's.

**What this does NOT mean:**

- Premature abstraction. Where v1.3 needs CFA-specific behavior to ship the first dimension, that behavior is implemented as a documented exception with an explicit "generalize when second tenant adopts" note. Speculative generality across all four dimensions on day one would slow delivery without proportionate value.
- Speculative connectors. Each connector is built for a real tenant's real data source. Building a generic "SharePoint connector that works for any tenant" before any second tenant uses it is YAGNI. Build for CFA first; harden for second tenant when Borderplex actually needs the dimension.
- Tenant-aware UI components. The cockpit is rendered server-side per tenant; UI components don't need tenant-awareness, only the data layer does.

**Tenant model:**

Per Gary's `wfdos-common` design (see discovery report dated 2026-04-23), tenants are represented as string slugs:
- `waifinder-flagship` (platform owner, internal use)
- `borderplex` (intermediary client)
- `cfa` (intermediary client — to be added in v1.3.0)

Both CFA and Borderplex fall under the **intermediary** segment. They differ from each other in operational details (grants administered, program scope, payment terms with Waifinder) but are architecturally equivalent. There is no "CFA-special" path in the codebase.

## 2. wfdos-common integration plan

v1.2 was built before `wfdos-common` existed in usable form. This section enumerates the modules in `wfdos-common` and the integration approach for each.

### 2.1 db — per-tenant SQLAlchemy engine factory

**v1.2 state:** Compliance engine creates its own SQLAlchemy engine via local connection-string assembly. No tenant scoping.

**v1.3 target:** Replace local engine creation with `wfdos_common.db.register_tenant("cfa", connection_url)` at startup. All session creation goes through `wfdos_common.db.session_scope(tenant_id="cfa")` (server-side) or `db_session(request)` FastAPI dependency (request-scoped).

**Migration approach:**
- Replace existing `get_db_session()` calls with `wfdos_common.db.session_scope(tenant_id=...)`
- Replace FastAPI dependency in routes with `db_session(request)` which reads `request.state.tenant_id`
- Remove local `engine.py` once all call sites migrated

**Open question:** Where does the per-tenant connection URL come from at startup? Hardcoded environment variables (one per tenant) is the simplest answer; a tenants table in a control-plane DB is the more scalable answer. Defer decision to v1.3.0 implementation; ask Gary what `wfdos-common` expects.

### 2.2 auth — magic-link + session middleware + role deps

**v1.2 state:** No auth on FastAPI routes. Anyone with network access to the engine or cockpit can call any endpoint.

**v1.3 target:** Install `wfdos_common.auth.SessionMiddleware` on both compliance engine and cockpit. Gate routes via `wfdos_common.auth.require_role("staff")` or appropriate ladder position. Mount `build_auth_router()` at `/auth/*` for magic-link login.

**Role mapping for compliance/cockpit access:**
- `admin` — Ritu (CFA), Waifinder platform admins
- `staff` — Krista (finance), Bethany (placements), Gage (developer with operational access)
- `workforce-development` — provisional, may not be needed for v1.3 surfaces
- `student` — explicitly NOT allowed access to compliance/cockpit

**Tier decorators:** routes that perform LLM calls should use `@llm_gated(...)` to give Gary's infrastructure the chance to enforce per-tier rate limits.

### 2.3 errors — typed APIError hierarchy

**v1.2 state:** Ad-hoc error responses. Some routes return `HTTPException`, some return JSON dicts with error fields, some let exceptions bubble.

**v1.3 target:** Replace ad-hoc patterns with `wfdos_common.errors.APIError` subclasses. Install `install_error_handlers(app)` on both services. Standardizes error response shape across all WFD-OS surfaces.

**Migration mapping (compliance engine):**
- `404` responses → `NotFoundError`
- `400` validation responses → `ValidationFailure`
- QB OAuth failures → `UnauthorizedError`
- Engine-internal failures with retry potential → `ServiceUnavailableError`

### 2.4 llm — provider-agnostic adapter with fallthrough

**v1.2 state:** Verdict generator (cockpit) and explanation endpoint (engine) call Gemini directly via `google.generativeai`. Bare provider, no fallthrough.

**v1.3 target:** Migrate both to `wfdos_common.llm.complete(messages, tier=..., system=...)`. Get Azure OpenAI as default with Anthropic and Gemini as fallthroughs. Provider failures become transparent — cockpit and engine no longer need to handle provider-specific errors.

**Migration scope:**
- `agents/finance/verdict_generator.py` — replace `genai.GenerativeModel(...).generate_content(...)` with `wfdos_common.llm.complete(...)`
- `agents/grant-compliance/.../explain_flag.py` (or wherever the explain endpoint lives) — same pattern
- Remove `google-generativeai` as a direct dependency once both call sites migrated; `wfdos-common` carries provider SDKs internally

**Tier selection:** Verdict generation is a low-stakes editorial task. Use `tier="default"` (cheap, fast). Explanation generation is similarly editorial. Reserve `tier="reasoning"` for any future workflows that need deeper LLM reasoning (e.g., draft-time-effort-certification generation in v1.3.3).

### 2.5 graph — MS Graph client (SharePoint)

**v1.2 state:** Compliance engine has its own SharePoint reader code in `agents/grant-compliance/.../graph_helper.py` (or similar). Pre-`wfdos-common`.

**v1.3 target:** Migrate to `wfdos_common.graph`. Per discovery report, this is largely a 1:1 import migration since `wfdos-common`'s Graph submodule is itself a migration of `agents/graph/sharepoint.py`.

**Concern:** `wfdos-common.graph` uses a single global `ClientSecretCredential(tenant_id=AZURE_TENANT_ID, ...)` — not per-tenant. For CFA-only deployment in v1.3, this works (CFA's SharePoint is the only one being read). For multi-tenant production, each tenant needs its own credentials.

**v1.3 decision:** Accept the single-global-credential limitation for v1.3.0 integration. Document as known constraint. Per-tenant Graph credentials become v1.4 work or a follow-up Gary discussion.

### 2.6 logging — structlog with request context

**v1.2 state:** Mixed `print()`, `logging.info()`, ad-hoc patterns.

**v1.3 target:** Adopt `wfdos_common.logging` for structured logs with request context (tenant_id, request_id, actor). Greatly improves observability for any production debugging.

### 2.7 tenancy — BrandConfig registry

**v1.3 target:** Register CFA as a brand at startup via `wfdos_common.tenancy.register_brand(BrandConfig(slug="cfa", display_name="Computing for All", ...))`. Document brand fields needed (display name, primary color, logo path if cockpit becomes brand-aware in UI).

### 2.8 agent — Agent ABC + ToolRegistry

**v1.3 decision:** Defer migration of existing assistant agents (`agents/assistant/finance_agent.py`, etc.) onto the Agent ABC pattern. Per discovery report, this migration is itself a follow-up in `wfdos-common` (#26). Compliance engine and cockpit don't currently expose conversational agents that would benefit from the ABC. Revisit in v1.4.

## 3. Tenant onboarding mechanics for CFA

CFA must be added as a tenant in `wfdos-common` before any v1.3 work can proceed. This section specifies what onboarding entails.

### 3.1 Registration steps

Required actions, in order:

1. **Provision per-tenant database.** A schema or database for CFA's compliance data. Approach depends on Gary's `wfdos-common.db` model — schema-per-tenant in a shared DB, or DB-per-tenant. Decision deferred to v1.3.0 implementation; ask Gary.
2. **Register tenant in `wfdos_common.db`.** Call `register_tenant("cfa", connection_url_for_cfa)` at compliance engine and cockpit startup.
3. **Register brand in `wfdos_common.tenancy`.** Call `register_brand(BrandConfig(slug="cfa", display_name="Computing for All", ...))`.
4. **Seed CFA-specific reference data:**
   - Grant K8341 record (per CFA's grants table once it exists)
   - GJC compliance supplement reference (program-specific compliance rules)
   - Subrecipient seed data (the providers — Ada, Year Up, Vets2Tech, etc.)
   - User-actor mapping (krista@cfa.org → role staff, ritu@computingforall.org → role admin, gage@... → role staff or admin per Ritu's call)
5. **Migrate v1.2 compliance data.** Existing flags, transactions, attachments need to land in CFA's tenant scope. One-time data migration script.

### 3.2 Tenant resolution at request time

Per `wfdos_common.db.middleware.TenantResolver` (the authoritative middleware per Gary's design):
- Tenant resolved from `X-Tenant-Id` HTTP header (set by nginx edge proxy at trust boundary), OR
- Host header → tenant mapping (e.g., `cfa.wfd-os.com` → tenant `cfa`), OR
- Default tenant via `WFDOS_DEFAULT_TENANT_ID` environment setting

For local CFA development: set `WFDOS_DEFAULT_TENANT_ID=cfa` in `.env`. For production: nginx routes per host header.

### 3.3 What "CFA as tenant" looks like to users

No user-visible change for CFA staff in v1.3. Krista and Bethany continue to see the cockpit and compliance engine as they did in v1.2; tenant scoping happens beneath the surface.

When Borderplex (or any other intermediary) becomes a tenant of the platform, their staff see their own data, their own brand, their own subrecipients. Same code path; different tenant scope.

## 4. Path 2 readiness debt resolution (from v1.2)

v1.2 was built with several CFA-specific shortcuts. Each must be parameterized for multi-tenancy in v1.3. This section enumerates them with the resolution approach.

### 4.1 Role-to-display-name mapping

**v1.2 state:** Hardcoded in `agents/finance/audit_dimension_display.py`:

```python
_ROLE_DISPLAY_NAMES = {"bookkeeper": "Krista", "executive_director": "Ritu", "program_operations": "Bethany · Gage"}
```

**v1.3 resolution:** Replace with per-tenant config lookup. Each tenant has a role-to-person mapping stored in their tenant config (likely as a JSON column on a `tenant_config` table or as part of the brand registration). Lookup signature becomes `display_name_for_role(role: str, tenant_id: str) -> str`.

### 4.2 Actor-to-display-name mapping

**v1.2 state:** Hardcoded in `agents/finance/audit_activity_labels.py`:

```python
_ACTOR_DISPLAY_NAMES = {"krista@cfa.org": "Krista", "ritu@computingforall.org": "Ritu", ...}
```

**v1.3 resolution:** Same pattern as 4.1. Per-tenant config maps actor identifiers (typically email addresses) to display names. Unknown actors fall back to local-part of email.

### 4.3 Hardcoded "K8341" references

**v1.2 state:** Several places assume the active grant is K8341. Documentation gap threshold ($2,500) is K8341-derived. T&E placeholder copy mentions K8341 explicitly.

**v1.3 resolution:** Introduce a `Grant` model in the compliance engine (per-tenant scoped). Per-tenant config specifies which grants are "active" for compliance purposes. Computations parameterize on grant or aggregate across all active grants for the tenant. Documentation gap threshold becomes a per-grant configuration.

### 4.4 GJC Compliance Supplement assumptions

**v1.2 state:** The 12 Subpart E rules in `unallowable_costs.py` are general 2 CFR 200 Subpart E. But GJC-specific compliance supplement requirements would also apply if surfaced.

**v1.3 resolution:** Introduce a `ProgramCompliance` reference data model. Each grant references a program (GJC for K8341, others for hypothetical other CFA grants or Borderplex grants). Program-specific compliance rules load per grant. Subpart E rules remain shared reference data.

### 4.5 Cockpit fallback dimension titles

**v1.2 state:** `_FALLBACK_DIMENSION_TITLES` in `cockpit_data.py` hardcodes the six audit dimensions for the engine-unreachable case.

**v1.3 resolution:** Cockpit fetches dimension metadata from engine on startup and caches it. Engine-unreachable fallback uses the cached metadata, not hardcoded literals. If the cockpit has never successfully reached the engine, falls back to a generic "Audit dimensions unavailable — engine has not yet been reached" message.

### 4.6 Test infrastructure gap on cockpit branch

**v1.2 state:** Cockpit branch has no pytest infrastructure. All cockpit-side test work was deferred.

**v1.3 resolution:** Set up pytest scaffolding on the cockpit branch (or its v1.3 successor) as part of v1.3.0 integration work. Backfill tests for the v1.2 cockpit-side modules (`audit_dimension_display.py`, `audit_activity_labels.py`, `verdict_generator.py`). Future cockpit work has tests.

### 4.7 No JS/TS test infrastructure

**v1.2 state:** No Jest/Vitest in `portal/student/package.json`.

**v1.3 resolution:** Lower priority than Python pytest. Defer to v1.4 unless Gage finds the gap blocks meaningful work.

## 5. Per-tenant credential vault for QB OAuth tokens

v1.2 deferred QuickBooks OAuth token encryption and refresh handling pending Gary's `wfdos-common` work. Discovery report confirms `wfdos-common` does NOT include a credential vault — magic-link and session tokens use `itsdangerous` HMAC signing, which is fine for auth cookies but inappropriate for tenant-credential storage.

The QB OAuth token storage problem remains. v1.3 must solve it.

### 5.1 Requirements

- **Encryption at rest.** OAuth refresh tokens are long-lived secrets that grant access to QuickBooks data. Cannot be stored plaintext.
- **Per-tenant isolation.** Tenant A's credentials must be cryptographically inaccessible from tenant B's request context. A breach of one tenant's session must not expose another tenant's credentials.
- **Refresh handling.** OAuth access tokens expire (typically 1 hour). Refresh tokens have longer lifetimes but also expire. Refresh logic must be automatic and transparent to calling code.
- **Audit trail.** Every credential access logs (tenant, actor, purpose, timestamp).

### 5.2 Architecture options

**Option A — Add to `wfdos-common`.** Cleanest for Path 2: every tenant gets the same credential primitive. Requires Gary's involvement and acceptance of the design. Likely the right long-term answer.

**Option B — Build inside grant-compliance, designed for promotion.** Build a `credentials.py` module inside the compliance engine that follows interfaces a future `wfdos-common.credentials` module would expose. When ready to promote, copy to `wfdos-common` with minimal API change.

**Option C — Live with plaintext through v1.3.** Document as known security debt. Defensible only if (a) production deployment doesn't yet exist, (b) database access is tightly controlled, (c) v1.4 commits to addressing.

**v1.3 recommendation:** Option B. Builds the primitive while keeping forward momentum on v1.3 integration. Gives Gary something concrete to react to if/when the primitive promotes to `wfdos-common`. Avoids blocking v1.3.0 on a Gary conversation.

### 5.3 Implementation sketch (Option B)

Module `agents/grant-compliance/.../credentials.py`:

- `class CredentialVault` — abstract or concrete; per-tenant scoped
- `vault.store_oauth_credentials(provider: str, access_token: str, refresh_token: str, expires_at: datetime, metadata: dict)` — stores encrypted; provider is e.g. `"qb_online"`
- `vault.get_oauth_credentials(provider: str) -> OAuthCredentials | None` — returns decrypted; raises `UnauthorizedError` if no creds stored or refresh fails
- `vault.refresh_oauth_credentials(provider: str) -> OAuthCredentials` — explicit refresh; called automatically by `get_oauth_credentials` if access token has expired

Encryption: Fernet (cryptography library), key per-deployment via environment variable. Future per-tenant keys would extend this.

Storage: new `tenant_credentials` table, scoped by `tenant_id` per the per-tenant DB model.

Audit: every store / get / refresh writes to the existing `audit_log` table (action types `credentials.store` / `credentials.access` / `credentials.refresh`).

## 6. Tenant middleware reconciliation (Gary dependency)

### 6.1 The issue

Per discovery report dated 2026-04-23, `wfdos-common` ships two middlewares that both write `request.state.tenant_id`:

- `wfdos_common.db.middleware.TenantResolver` — established as authoritative per Gary's design (PR #22a)
- `wfdos_common.tenancy.TenantResolutionMiddleware` — also sets `request.state.tenant_id` AND `request.state.brand`

No reconciliation between them. A service installing both resolves tenants twice. A service installing only one misses either tenant resolution (brand-only middleware) or brand resolution (db-only middleware).

Brand resolution is handled separately on the cockpit by reading the resolved tenant_id and looking up the brand via `wfdos_common.tenancy.get_brand(tenant_id)` in the request handler — bypassing the second middleware.

This decision is provisional. Gary should confirm it matches his intent. If Gary wants both middlewares installed (or neither, in favor of a unified third option), v1.3 adopts whatever he specifies. Do not block v1.3 implementation on the conversation — proceed with this provisional choice and adjust if Gary corrects it.

## 7. Connector framework

The four placeholder dimensions (subrecipient monitoring, performance reporting, time and effort, procurement) all need a way to populate per-dimension data into the compliance engine from external sources. This section defines the framework all four use.

### 7.1 Design principles

- **Narrow responsibility.** A connector is a data fetcher. It reads from an external source (SharePoint, payroll system, WSAC portal export, contract folder) and writes to the engine's tenant-scoped tables. It does NOT compute compliance readiness percentages or gap lists — those remain in `dimension_readiness.py` and consume the populated data.
- **Tenant-scoped.** Every connector operation is scoped to a single tenant. A connector is instantiated with a tenant context (typically a session bound to that tenant's database scope) and never crosses tenant boundaries.
- **Idempotent.** Running a connector twice produces the same end state as running it once. New data is added; existing data is updated; nothing is duplicated. Connectors track sync state to support resumable operation.
- **Audited.** Every connector run writes to the existing `audit_log` table. Action types follow the convention `connector.{name}.{action}` (e.g., `connector.subrecipient.sync_start`, `connector.subrecipient.sync_complete`, `connector.subrecipient.error`).
- **Failure-tolerant.** A connector failure does not corrupt existing data. Partial syncs leave the engine in a consistent state; the next sync resumes cleanly. Errors are logged but do not crash the engine.

### 7.2 Connector interface

Abstract base class in `agents/grant-compliance/.../connectors/base.py`:

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass
from datetime import datetime
from sqlalchemy.orm import Session

@dataclass
class ConnectorRunResult:
    connector_name: str
    tenant_id: str
    started_at: datetime
    completed_at: datetime
    records_added: int
    records_updated: int
    records_skipped: int
    errors: list[str]

    @property
    def success(self) -> bool:
        return len(self.errors) == 0

class Connector(ABC):
    """Base class for all dimension data connectors.

    Subclasses are responsible for reading from an external source,
    normalizing data, and writing to the engine's tenant-scoped tables.
    They do not compute compliance percentages.
    """

    name: str  # connector slug, used in audit log action types

    def __init__(self, session: Session, tenant_id: str):
        self.session = session
        self.tenant_id = tenant_id

    @abstractmethod
    def fetch(self) -> ConnectorRunResult:
        """Read from external source, write to engine tables.

        Must be idempotent. Must write to audit_log on start, completion,
        and any errors. Must not crash on individual record failures —
        continue processing and record errors in the result.
        """
        ...

    def is_due(self) -> bool:
        """Returns True if this connector should run now.

        Default implementation always returns True. Override for time-
        based scheduling (e.g., 'only run if last successful sync was
        more than 24 hours ago').
        """
        return True
```

### 7.3 Two execution modes

Connectors support two execution patterns:

**Mode A — Scheduled fetch.** Connector runs on a regular cadence (typically daily). Reads from external source, syncs new/changed data into engine tables. Examples: SharePoint folder scan for subrecipient documents, WSAC portal export ingestion.

Scheduling mechanism is deferred to implementation. Likely options include: a separate scheduler service in wfd-os, cron jobs invoking management commands, or per-request lazy refresh with `is_due()` gates. Gage to choose based on what wfd-os deployment infrastructure provides.

**Mode B — User-triggered.** Connector runs in response to a user action (e.g., Krista submits a manual subrecipient risk assessment via a form, triggering immediate ingestion). Same `Connector.fetch()` interface; just invoked synchronously from a request handler instead of from a scheduler.

### 7.4 Connector registry

Connectors register at startup via a registry pattern:

```python
# agents/grant-compliance/.../connectors/registry.py
_CONNECTORS: dict[str, type[Connector]] = {}

def register_connector(cls: type[Connector]) -> type[Connector]:
    _CONNECTORS[cls.name] = cls
    return cls

def get_connector(name: str) -> type[Connector] | None:
    return _CONNECTORS.get(name)

def all_connectors() -> dict[str, type[Connector]]:
    return dict(_CONNECTORS)
```

Each connector module decorates its class with `@register_connector`. Registry enables generic invocation patterns (e.g., a "run all due connectors" management command) without hardcoding the connector list.

### 7.5 Per-tenant configuration

Each connector reads its configuration from per-tenant config (the same per-tenant config introduced in §4.1-4.4 for role mappings, grant references, etc.). Configuration keys follow the convention `connector.{name}.{field}`. Examples:

- `connector.subrecipient.sharepoint_folder_path` — path to subrecipient parent folder for the tenant
- `connector.performance_reporting.wsac_export_folder` — path to WSAC portal exports
- `connector.time_effort.payroll_source` — identifier for the tenant's payroll integration approach

Configuration absent for a tenant means that connector is not configured for that tenant; running it returns a `ConnectorRunResult` with an explanatory error and zero records changed. The dimension this connector feeds shows the existing "Readiness measurement not yet available — connector not configured for tenant" placeholder message.

### 7.6 Relationship to compliance computation

Connectors populate data. `dimension_readiness.py` computes readiness from populated data. The two layers communicate only via the database — connectors do not invoke compute functions; compute functions do not invoke connectors.

When a dimension's connector has been configured and run successfully for a tenant, the dimension transitions from `status: "placeholder"` to `status: "computed"` automatically (because the underlying tables now have data, and compute functions return values instead of `None`).

When a dimension's connector has been configured but never successfully run, or when configured but most recent run had errors, the dimension shows `status: "computed"` with `readiness_pct: null` (per the three-state status model from v1.2 spec §Three-state dimension status). The drill panel shows the most recent connector error so the operator knows what to investigate.

### 7.7 Testing

Each connector subclass has unit tests covering:

- Happy path: fetch from a mocked external source, verify correct records written
- Idempotency: run fetch twice, verify second run produces zero added records
- Error handling: mock external source to raise; verify `result.errors` populated, no partial corruption
- Audit log writes: verify action types written on start, completion, and error
- Tenant scoping: verify writes go to the correct tenant scope, no leakage

### 7.8 Implementation note for Gage

The connector framework is built once in v1.3.0 (alongside wfdos-common integration) and used four times in v1.3.1-v1.3.4. Each dimension's connector is its own focused build. The framework should be deliberately minimal at first — additional features (parallelism, distributed locking, retry policies) become real work when concrete needs surface, not speculatively.

## 8. v1.3.1 — Subrecipient monitoring

### 8.1 Compliance regulation

Subrecipient monitoring requirements live in 2 CFR 200.331-333:

- **§200.331 — Requirements for pass-through entities.** When a non-Federal entity (e.g., CFA) passes Federal funds through to subrecipients (e.g., training providers under K8341), the pass-through entity is responsible for ensuring the subaward complies with Federal statutes and regulations and the terms of the subaward. Specific requirements include making subrecipient determinations vs. contractor determinations, providing required identifiers in subaward documents, and maintaining records.
- **§200.332 — Requirements for pass-through entities (continued).** Pass-through entities must evaluate each subrecipient's risk of noncompliance, monitor activities to ensure subaward compliance, verify subrecipient audits per Subpart F when applicable, and consider whether the results of monitoring necessitate adjustments.
- **§200.333 — Fixed amount subawards.** Specific rules when subawards are fixed-amount; relevant to CFA's per-placement payment structure for training providers.

**What auditors look for during a Single Audit:**

- Documented risk assessment for each subrecipient at award and updated periodically
- Monitoring activities (site visits, desk reviews, financial monitoring) appropriate to assessed risk
- Records of monitoring findings and follow-up
- Verification that subrecipients with $1M+ in Federal awards have completed their own Single Audit and that any findings affecting CFA's funds have been resolved
- Documentation that monitoring frequency and depth was risk-proportionate

**Why this matters for nonprofits generally:**

Subrecipient monitoring is consistently among the top finding categories in nonprofit Single Audits. Auditors specifically test that risk assessments exist, are documented, and demonstrate genuine assessment (not pro-forma checkbox exercises). For CFA's K8341, the subrecipient universe consists of the strategic partners — WABS, WTIA, Evergreen Goodwill, Seattle Jobs Initiative, ESD 112, I&CT, and NCESD — that carry out programmatic activities for the K8341 program. CFA has determined per §200.331(b) that training providers (Ada, Year Up, Vets2Tech, Per Scholas, Code Day, Apprenti), Riipen, and the CFA Contractors line items (AI Engage, Pete & Kelly Vargo) are contractors rather than subrecipients; their compliance mechanism is the contractor verification approach captured in §11 (procurement and competition).

3. For each subrecipient folder, scan contents for documents matching expected types. File naming conventions infer document type and date.

### 8.5 Compute function

Function: `compute_subrecipient_monitoring(db, tenant_id) -> int | None` in `dimension_readiness.py`.

**Formula:** Weighted composite of three factors, returned as an integer percentage.

Factor 1 — Risk assessment coverage (40% weight):
- Numerator: count of active subrecipients with a risk assessment dated within the last 12 months
- Denominator: count of active subrecipients

Factor 2 — Monitoring activity coverage (40% weight):
- For each active subrecipient, expected monitoring frequency depends on risk level:
  - High risk: monitoring activity in the last 6 months
  - Moderate risk: monitoring activity in the last 12 months
  - Low risk: monitoring activity in the last 24 months
- Numerator: count of active subrecipients meeting their risk-tier monitoring expectation
- Denominator: count of active subrecipients with a current risk assessment (subrecipients without risk assessments are not counted in this factor since their expected frequency is unknown)

Factor 3 — Audit verification (20% weight):
- For subrecipients with `federal_award_threshold_exceeded = true`:
  - Numerator: count with audit report received and `all_findings_resolved = true` for findings affecting pass-through funds
  - Denominator: count of subrecipients above the threshold
- If no subrecipients are above the threshold, this factor is N/A and weights redistribute (50/50 between factors 1 and 2)

Composite formula: `100 * (0.4 * factor_1 + 0.4 * factor_2 + 0.2 * factor_3)` where each factor is a ratio in [0, 1]. Round to integer. Apply `_clamp_pct` per v1.2 spec §Defensive percentage clamping.

**Initial readiness expectation (v1.3.0-draft.9 update):** Initial computed readiness for CFA's K8341 strategic partner subrecipients is expected to be substantially high given Alka has confirmed monitoring is complete for all seven subrecipients (WABS, WTIA, Evergreen Goodwill, Seattle Jobs Initiative, ESD 112, I&CT, NCESD). Once Alka's documentation is ingested, factor 1 (risk assessment coverage) and factor 2 (monitoring activity coverage) should reflect this completed work. Factor 3 (audit verification) depends separately on whether subrecipient audit reports are available — applies to subrecipients with $1M+ Federal awards in their fiscal year. Initial readiness percentage is a useful sanity check on connector implementation: if it produces a number significantly out of line with Alka's confirmed monitoring completeness, the data ingestion or compute function likely has a defect.

**Edge cases:**

- No subrecipients configured for tenant: return `None`. Dimension shows `status: "computed"` with `readiness_pct: null` and drill panel says "No subrecipients configured. Add subrecipients via the cockpit subrecipient management view." (Per the three-state status model — formula exists, no data yet.)
- Subrecipients exist but no risk assessments: factor 1 numerator is 0, contributing 0 to composite. Factor 2 denominator is 0, factor 2 not computed. Factor 3 may still compute if audit data exists. Composite uses available factors with proportionally redistributed weights, OR returns `None` if too sparse to be meaningful (decision: if factor 1 returns 0 AND factor 2 has zero denominator AND factor 3 has zero denominator, return `None`).
- Connector has never run successfully: same as no subrecipients configured.

### 8.6 Gap detection

Function: `gaps_for_subrecipient_monitoring(db, tenant_id) -> list[dict]` in `dimension_readiness.py`.

A gap is one of these conditions per active subrecipient:

- Missing risk assessment (no assessment exists) — severity: `blocker`
- Stale risk assessment (last assessment > 12 months ago) — severity: `warning`
- Insufficient monitoring activity for risk tier (per §8.5 thresholds) — severity: depends on tier (`blocker` for high-risk, `warning` for moderate, `info` for low)
- Missing audit report for above-threshold subrecipient — severity: `blocker`
- Audit findings affecting pass-through funds not resolved — severity: `blocker`
- Active subrecipient with `subaward_status = terminated_for_cause` lacking documented monitoring of contributing factors — severity: `warning`

Each gap entry per the v1.2 gap shape (§Change 2 of the audit readiness tab spec):
- `type`: `"subrecipient_monitoring_gap"`
- `id`: composite of subrecipient_id + gap_type
- `title`: e.g., "Ada Developers Academy — Risk assessment missing"
- `detail`: explanation of what's missing and the relevant CFR citation
- `metadata`: `{subrecipient_id, subrecipient_display_name, gap_type, severity, last_relevant_date}`

Sorted by severity desc (blocker > warning > info), then by subrecipient_display_name asc.

### 9.1 Compliance regulation

## 10. v1.3.3 — Time and effort certifications

**Scope note:** This dimension is the largest and most workflow-heavy of the four v1.3 dimensions. Unlike subrecipient monitoring (§8) where existing documentation needs ingestion, or performance reporting (§9) where existing operational work needs structured capture, time and effort certifications require building a quarterly attestation workflow that does not currently exist at CFA. This section specifies both the data model and the workflow. Implementation is correspondingly larger; sequence accordingly per §14.

### 10.1 Compliance regulation

Time and effort documentation requirements live in 2 CFR 200.430(i):

- **§200.430(i)(1).** Compensation for personal services charged to Federal awards must be based on records that accurately reflect the work performed. Records must be supported by a system of internal controls that provides reasonable assurance that the charges are accurate, allowable, and properly allocated.
- **§200.430(i)(1)(i)-(viii).** Records must: be supported by an effective internal control system; reasonably reflect total activity for which the employee is compensated; encompass both Federally-assisted and other activities; comply with established accounting policies; support distribution of salary or wages among specific activities or cost objectives; not be required for time spent solely on a Federal award when the activity is documented in approved budget; budget estimates alone do not qualify as support; significant changes must be identified and entered into the records in a timely manner.
- **§200.430(i)(2).** For records that meet these standards, the non-Federal entity will not be required to provide additional support or documentation for the work performed.

**What auditors look for during a Single Audit:**

- Signed certifications from employees performing work charged to Federal awards
- Certifications cover the work period being audited (typically quarterly or semi-annually)
- Certifications reflect actual time/effort distribution, not just budgeted amounts
- System of internal controls demonstrating accuracy (e.g., supervisor review, periodic reconciliation against payroll records)
- Significant changes to effort distribution captured in timely amendments
- Charges to Federal awards reconcile to certified effort percentages

**Why this matters for K8341 specifically:**

The named roles in the K8341 budget — Project Director, Project Manager, Finance Manager, Data Manager, WBL Portal Director, Education & Industry Engagement Manager — represent CFA staff whose compensation is partially or fully charged to the grant. §200.430(i) requires documented certification of the effort each of these individuals devotes to K8341 vs. other activities. Without such certifications, the salary charges to K8341 are at risk during audit (potentially leading to questioned costs and required refunds).

**Why this matters for nonprofits generally:**

Time and effort is consistently the highest-finding dimension in nonprofit Single Audits. Auditors specifically test that certifications exist, are signed by the employee (not just supervisor), reflect actual rather than budgeted distribution, and align with payroll charges. Nonprofits frequently fail this dimension because: (a) no formal certification process exists; (b) the process exists but isn't followed consistently; (c) certifications are signed but reflect the budget, not actual effort. The remediation cost can be material — questioned costs may need to be refunded to the funder.

**Current CFA state:**

CFA does not currently have a formal quarterly time and effort certification workflow. This dimension's readiness in v1.2 is appropriately shown as a placeholder pending v1.3.3 implementation. The implementation requires both a workflow build (notify employees, capture attestations, store signed records) AND ingestion of any existing payroll allocation data that may be partially relevant.

### 10.2 Data sources

The data needed to compute time and effort readiness comes from two sources, one existing-but-unreliable, one to-be-built:

**Existing source — QuickBooks Payroll project-based accounting.** CFA uses QuickBooks for payroll, which supports project-based time allocation. Per Ritu's input as of v1.3.0-draft.6, "it does have project-based accounting, but I am not sure how clean it is." This is a potential data source for *charge allocation* (which grants payroll dollars were charged to) but is NOT a substitute for §200.430(i) certifications, which require *employee-signed attestation* of actual effort. The QB data informs the audit reconciliation step (do certified effort percentages align with actual payroll charges) but does not satisfy the certification requirement itself.

**To-be-built source — quarterly attestation workflow.** v1.3.3 includes building a workflow where each federally-funded employee receives a quarterly notification, attests to their effort distribution across grants and non-grant activities for the prior quarter, and signs the certification. Records are stored as structured certifications in the compliance engine. This workflow is the audit-defensible source per §200.430(i).

**Tenant-specific variations to anticipate:**

Other intermediary tenants will have different payroll systems (Paycom, Gusto, ADP, etc.), different staff structures, and potentially different certification frequencies (semi-annual is also acceptable per §200.430(i)). Connector and workflow parameterize on per-tenant configuration.

**`employees`** — one row per employee subject to time and effort certification.
- `id` — UUID, primary key
- `tenant_id` — string

### 10.4 Connector and workflow implementation

v1.3.3 has two implementation components: an attestation workflow and a payroll reconciliation connector.

#### 10.4.1 Attestation workflow

**Workflow name:** `quarterly_time_effort_attestation`

**Trigger:** Scheduled — runs at the end of each calendar quarter (or per tenant configuration). For CFA, quarters align with calendar (Q1 ends March 31, etc.). Some tenants may align with their fiscal year; configure per tenant.

**Steps:**

1. **Identify federally-funded employees.** Query `employees` where `federally_funded = true` and `end_date` is null or after the period end.
2. **Create pending certification records.** One `time_effort_certifications` row per identified employee, with status `pending`, period_start and period_end matching the closed quarter.
3. **Send notifications.** Email each employee with a link to the attestation form. Update certification status to `notified`, record `notification_sent_at`. Use `wfdos_common.email` for sending.
4. **Employee fills attestation form.** Web form (cockpit-side) where employee enters effort percentages per grant they're assigned to plus non-grant effort (must total 100%). Saving as draft updates status to `drafted`.
5. **Employee signs.** Final submission with explicit signature affirmation (checkbox + electronic signature equivalent, or upload of paper form). Status updates to `signed` (or `signed_late` if past quarter-end + grace period). Records `signed_at`, `signer_actor`, `signature_method`.
6. **Reminders.** If status is `notified` or `drafted` past a configurable threshold (e.g., 14 days from notification), send reminder. After hard cutoff (e.g., 30 days), status updates to `missed` and a compliance flag is raised.
7. **Audit log writes** per §7.1: `time_effort.notification_sent`, `time_effort.draft_saved`, `time_effort.certified`, `time_effort.missed`.

**Per-tenant configuration:**
- `attestation.cadence` — `quarterly` (default) or `semi_annual`
- `attestation.notification_template` — email body template, supports tenant branding
- `attestation.reminder_days` — days from notification to first reminder
- `attestation.hard_cutoff_days` — days from notification before status becomes `missed`
- `attestation.grace_period_days` — days past period_end where signing still counts as `signed` (vs. `signed_late`)

**Out of scope for v1.3.3:** mobile app, complex multi-grant routing logic, supervisor approval flows. Initial implementation is web-based attestation by employee directly. Supervisor approval can be added in v1.4 if audit feedback indicates it's needed.

#### 10.4.2 Payroll reconciliation connector

**Connector name:** `payroll_reconciliation`

**Connector class:** `PayrollReconciliationConnector(Connector)` per the framework in §7.2.

**External source:** QuickBooks Payroll project-based accounting reports (read via QB API using the credential vault from §5).

**Sync cadence:** Mode A — runs after each quarter closes, after the attestation workflow completes for that quarter.

**Per-tenant configuration:**
- `connector.payroll_reconciliation.qb_company_id` — tenant's QuickBooks company identifier
- `connector.payroll_reconciliation.qb_grant_class_mapping` — how QB classes (or projects) map to grant_id values in the compliance engine
- `connector.payroll_reconciliation.variance_threshold_pct` — percentage threshold above which a variance flags as needing attention (default 10%)

**Connector behavior:**

1. For each completed certification in the just-closed period, fetch the corresponding employee's payroll charges from QuickBooks for the same period.
2. Compute actual charge percentage per grant from QB data (using the qb_grant_class_mapping configuration).
3. Create or update a `payroll_charge_reconciliations` row linking certification to actual charges.
4. Compute variance per grant (certified pct minus actual pct). If absolute variance exceeds threshold, write a compliance flag.
5. Audit log writes: `connector.payroll_reconciliation.sync_start`, `connector.payroll_reconciliation.variance_flagged`, `connector.payroll_reconciliation.sync_complete`, `connector.payroll_reconciliation.error`.

**QB data quality caveat:** Per Ritu's input as of v1.3.0-draft.6, the cleanliness of CFA's QuickBooks project-based accounting is uncertain. v1.3.3 implementation should include a diagnostic step (run by Gage during build) to assess data quality before relying on it. If data is too unreliable, the reconciliation step degrades to "advisory only" — the certification itself remains the audit-defensible record, and reconciliation flags are informational rather than authoritative.

### 10.5 Compute function

Function: `compute_time_effort(db, tenant_id) -> int | None` in `dimension_readiness.py`.

**Formula:** Weighted composite of two factors, returned as integer percentage.

Factor 1 — Certification completion (70% weight):
- For each closed period (quarter) within the lookback window (default: last four closed quarters):
  - Numerator: count of certifications with status `signed` or `signed_late`
  - Denominator: count of certifications expected for that period (one per federally-funded employee active during that period)
- Aggregate across the lookback window.

Factor 2 — Reconciliation accuracy (30% weight):
- For each certification with completed payroll reconciliation:
  - Numerator: count of grant lines where absolute variance is below threshold
  - Denominator: total count of grant lines reconciled
- Aggregate across the lookback window.
- If no reconciliations exist (e.g., QB integration not yet active or data quality issue), this factor is N/A and weights redistribute (factor 1 takes 100%).

Composite: `100 * (0.7 * factor_1 + 0.3 * factor_2)` with redistribution if factor 2 is N/A. Round to integer. Apply `_clamp_pct`.

**Edge cases:**

- No federally-funded employees configured: return `None`. Drill panel shows "No federally-funded employees configured for tenant. Add employees and grant assignments via the cockpit."
- Federally-funded employees exist but no certifications yet (e.g., workflow newly deployed, first quarter not yet closed): return `None`. Drill panel shows "Certification workflow active. Awaiting first quarter completion."
- First quarter closed but no certifications signed: factor 1 returns 0, composite returns 0%. This is the correct state — readiness is genuinely 0% until employees actually sign.

### 10.6 Gap detection

Function: `gaps_for_time_effort(db, tenant_id) -> list[dict]` in `dimension_readiness.py`.

A gap is one of these conditions:

- Federally-funded employee has missing certification for a closed quarter (status `missed` or no record) — severity: `blocker`
- Federally-funded employee has unsigned certification past hard cutoff — severity: `blocker`
- Federally-funded employee has unsigned certification within cutoff window but past period end (status `notified` or `drafted` with period_end > 14 days ago) — severity: `warning`
- Certification signed but variance with payroll charges exceeds threshold and is unresolved — severity: `warning`
- Federally-funded employee record missing required fields (no role, no grant assignment for active period) — severity: `info`

Each gap entry per the v1.2 gap shape:
- `type`: `"time_effort_gap"`
- `id`: composite of employee_id + period + gap_type
- `title`: e.g., "Krista Lawrence — Q1 2026 certification missing"
- `detail`: explanation of what's missing and CFR citation (§200.430(i))
- `metadata`: `{employee_id, employee_display_name, period_start, period_end, gap_type, severity, days_overdue}`

Sorted by severity desc, then by days_overdue desc, then by employee_display_name asc.

### 11.1 Compliance regulation

Procurement standards live in 2 CFR 200.317-327:

- **§200.317 — Procurements by states.** N/A for CFA (a non-state recipient); standards in §200.318-327 apply instead.
- **§200.318 — General procurement standards.** Recipients must use documented procurement procedures consistent with state/local laws and the standards of §200.319-327. Conflicts of interest avoided. Records sufficient to detail procurement history maintained.
- **§200.319 — Competition.** Procurements must be conducted in a manner providing full and open competition consistent with §200.320 standards. Specific situations restricting competition (e.g., unreasonable requirements, organizational conflicts of interest) prohibited.
- **§200.320 — Methods of procurement.** Five recognized methods: (a) micro-purchase ($10,000 threshold or as adjusted), (b) small purchase ($250,000 threshold), (c) sealed bids (formal advertising), (d) competitive proposals, (e) noncompetitive proposals (sole source justification required).
- **§200.321 — Contracting with small and minority businesses, women's business enterprises, and labor surplus area firms.** Affirmative steps required to ensure such firms are used when possible.
- **§200.322 — Domestic preferences for procurement.** Build America requirements where applicable.
- **§200.323 — Procurement of recovered materials.** Energy and water-efficient procurement requirements.
- **§200.324 — Contract cost or price.** Cost and price analysis required for procurements above the simplified acquisition threshold.
- **§200.325 — Federal awarding agency or pass-through entity review.** Reviewable by funder.
- **§200.326 — Bonding requirements.** For construction or facility improvements.
- **§200.327 — Contract provisions.** Required provisions in contracts (e.g., termination clauses, equal employment opportunity).

**What auditors look for during a Single Audit:**

- Documented procurement procedures consistent with §200.318
- Procurement method selected appropriate to estimated dollar value (§200.320)
- Competition documented for procurements above micro-purchase threshold
- Sole source / noncompetitive procurements have written justification per §200.320(e)
- Cost or price analysis for procurements above simplified acquisition threshold (§200.324)
- Affirmative steps for small/minority/women's business inclusion documented (§200.321)
- Required contract provisions present in actual executed contracts (§200.327)
- Conflict of interest disclosures and resolutions documented (§200.318)

**Why this matters for K8341 specifically:**

CFA's K8341 procurement universe is structured but documented across multiple methods:

Kelly: $2,000/month retainer. Engagement basis (small purchase per §200.320(a) likely; specific procurement memo to be located by Gage with Krista).

**Why this matters for nonprofits generally:**

Capturing existing artifacts in a structured compliance system substantially reduces audit risk regardless of whether the underlying decisions were correct.

### 11.5 Compute function

Function: `compute_procurement(db, tenant_id) -> int | None` in `dimension_readiness.py`.

**Formula:** Weighted composite of three factors, returned as integer percentage.

Factor 1 — Procurement documentation completeness (50% weight):
- For each active or recently-closed contractor:
  - Required documents per procurement method (e.g., competitive proposals require RFP + evaluation + selection memo + executed contract; sole source requires sole source justification + executed contract)
  - Numerator: count of contractors with all required documents on file
  - Denominator: count of contractors in scope

Factor 2 — Cost/price analysis coverage (20% weight):
- For contractors above the simplified acquisition threshold:
  - Numerator: count with cost/price analysis on file
  - Denominator: count above threshold
- If no contractors exceed threshold, factor is N/A and weights redistribute.

Factor 3 — Operational compliance verification cadence (30% weight):
- For each active contractor:
  - Numerator: count where most recent `contractor_compliance_verifications` entry is within expected cadence (e.g., monthly invoice cycle for AI Engage and Vargos; per-payment for training providers)
  - Denominator: count of active contractors

Composite: `100 * (0.5 * factor_1 + 0.2 * factor_2 + 0.3 * factor_3)` with redistribution if factor 2 is N/A. Round to integer. Apply `_clamp_pct`.

**Edge cases:**

- No contractors configured for tenant: return `None`. Drill panel shows "No contractors configured. Add contractors via the cockpit procurement management view."
- Contractors exist but no procurement records ingested: factor 1 returns 0 (or low), factor 3 may compute if verifications exist independently. Composite uses available data with appropriate weighting redistribution.
- Connector has never run successfully: same as no contractors configured.

## 16. Open product decisions

Decisions where Ritu (product owner) is the deciding voice. Listed as they surface during spec authoring. Each is resolved via a spec amendment when decided.

### 16.1 Prior-format records ingestion policy (subrecipient monitoring)

**Context:** Alka has existing monitoring documentation for K8341 subrecipients in a format yet to be determined. Per §8.4, two scenarios are possible: structured ingestion as-is, or re-capture into the structured §8.3 data model.

**Decision required:** Policy for handling prior-format records when the structured data model differs from the existing format.

**Options:**
- (a) Ingest as-is with metadata noting source format. Preserves Alka's effort exactly. Compliance engine carries some non-canonical records that may need future cleanup.
- (b) Re-capture into structured form. Cleaner compliance engine state. Adds work for Alka or designee.
- (c) Hybrid: ingest essential fields as-is for immediate dimension computation, schedule re-capture for completeness over time.

**Considerations:** Audit defensibility (auditor wants to see records, not transformed records); operational burden on Alka; long-term cleanliness of compliance engine; precedent for similar situations in other dimensions or other tenants.

**Status:** Open. Resolution pending Alka's documentation being located and reviewed.

### 16.2 §200.331(b) determination documentation

**Context:** Per v1.3.0-draft.7 (with NCESD classification corrected in v1.3.0-draft.9), CFA has operationally classified K8341 providers per §200.331(b):

- **Subrecipients (7):** WABS, WTIA, Evergreen Goodwill, Seattle Jobs Initiative, ESD 112, I&CT, NCESD. Carry out programmatic activities for the K8341 program. NCESD is a hybrid case — primarily subrecipient, with some placement-related deliverables — but classified as subrecipient based on the substance of the relationship per §200.331(c).
- **Contractors (10):** Training providers (Ada, Year Up, Vets2Tech, Per Scholas, Code Day, Apprenti), Riipen, AI Engage, Pete Vargo, Kelly Vargo. Provide goods or services to CFA's program operations.

The determinations appear sound under the regulation, but auditors will expect them to be documented per-provider, not just operationally followed. NCESD specifically warrants careful documentation given its hybrid characteristics and the operational events around its termination.

**Decision required:** Whether to formally document the §200.331(b) determination for each provider as a v1.3 deliverable (creating per-provider memoranda explaining the contractor vs. subrecipient classification rationale), or to defer formal documentation to v1.4 with a documented acknowledgment that operational classification is the current basis.

**Options:**
- (a) Generate per-provider §200.331(b) determination memoranda as part of v1.3.4 (procurement) for contractors and v1.3.1 (subrecipient monitoring) for subrecipients. Each memo cites the relevant characteristics under §200.331(a)/(b) that support the classification. NCESD's memo specifically addresses the hybrid nature and the §200.331(c) judgment basis. Stronger audit defense; modest additional work; can be assisted by LLM with human review per memo.
- (b) Defer formal documentation to v1.4. v1.3 captures classification as data fields in the procurement_records and subrecipients tables, with operational rationale noted in free-text fields. Faster delivery of v1.3; carries an explicit known-gap risk for early-tenure audit testing.
- (c) Audit-firm-led approach. Defer the documentation decision until the audit firm engaged for the FY24-25 Single Audit provides their preference on documentation depth. Some firms accept structured data fields; others require formal memoranda.

**Considerations:** Audit defensibility under §200.331; firm-specific preferences; CFA staff workload to author memoranda (Ritu plus Krista or Alka); precedent for future tenants (Borderplex will face the same classification decisions); risk of the audit firm raising classification as a finding if documentation is weak; specific scrutiny NCESD's hybrid classification may attract during audit fieldwork.

**Status:** Open. Resolution likely requires audit firm engagement to inform Option C path. Could be partially resolved earlier by Ritu making a Path A or B call independently.
